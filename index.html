Go 发布二进制静态库的方法

众所周知，Go的编译系统是针对源代码的，如果想发布一个二进制文件给别人，该怎么做？如何在CI上编译不同平台的C代码？

下面跟我一起来实验下：

  1. 创建好项目
    在src目录下创建两个包路径，liba 和 main，liba用来实验隐藏代码
  1. 项目各文件如下

  • liba/dummy.go, 调用代码先注释掉，避免冲突

  • liba/real.go

  • main/main.go

  1. 好了，我们先编译真实的library, 得到.a 文件，并跑下看看

  2. 将真实的library 拷贝出来备用
    cp pkg/darwin_amd64/liba.a .

  3. 将real.go 的代码注释掉，dummy.go 的代码取消注释，并编译跑下看看

  4. 见证奇迹的时刻到了，我们把real.go 的实现给copy回来，编译跑下看看

通过这个实验，可以看出go最后的链接步骤是比较代码和.a文件的创建时间，如果.a 创建的晚，就不会再次对代码进行编译，只做链接。大家可以将上面的cp 换成mv验证下 :) 总结下，如果想发布二进制的go library，就要将.a 文件和 stub 代码一起交付给使用者，并且在编译的时候做一次替换。

如果有些软件需要跨平台编译，而且还有些底层访问的C代码，可以先编译成.a 文件，然后放到CI 平台上，让Go交叉编译的时候链接就好了。

Go代码风格检查

什么是style

  • 几乎所有讲代码书都会谈到一个组织内要遵循同样的code style。这个style很多公司都有制定相应的规范。C style，java style等等。内容包括但不限于代码行长度，函数如何命名等等，这些非编译器及代码质量相关工具检查范围。

为什么要进行style check

  • 好的style 是为人服务，而不是为机器服务，因为已经过了机器那一关。他必须使代码易于阅读，易于修改，易于理解，易于传播，易于沟通。比如，一行代码过长就不利于阅读，函数命名过长或者过简略都不好，文件行数过多不利于多人同时修改等等。

为什么需要用工具进行检查

  • 肉眼很难检查是否做了统一的格式化。
  • 让review者去数代码行数或者字数太浪费时间。
  • review者对命名进行检查挺烦也会有遗漏。

工具的要求

  • 因为每个组织或个人的角度不一样，所以要是可定制的
  • 尽量覆盖常见的code style.
  • 可以忽略部分代码

常见的style check工具

  • Python: PEP8, flake8,
  • Ruby: RuboCop
  • PHP: CodeSniffer
  • Java: CheckStyle

一些网络风格检查服务，可以评分以及提供修改意见

  • codeclimate.com，支持Python, Ruby, PHP, JavaScript
  • scrutinizer-ci.com，支持PHP, Ruby, Python

下面进入正题，Go Check Style

  • 我参考Java Check Style，以及Go Code Review Comments 进行编写
  • 可以检查参数及返回值个数
  • 可以检查是否格式化
  • 可以检查包名是否符合规范
  • 可以检查函数行数以及文件行数
  • 直接使用json进行配置，可以整合到jenkins里

项目地址

https://github.com/qiniu/checkstyle
